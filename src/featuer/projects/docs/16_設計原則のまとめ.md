# 16: 設計原則のまとめ 🎓

## 結論：設計原則のまとめ

```
【設計思考の順序（プロジェクト機能）】

1. ドメイン分析 ✅
   └─ projectsテーブル → Project Managementドメイン

2. ユースケース抽出 ✅
   └─ 6つのユースケースを定義
      ├─ UC-01: プロジェクト作成
      ├─ UC-02: プロジェクト一覧取得
      ├─ UC-03: プロジェクト詳細取得
      ├─ UC-04: プロジェクト更新
      ├─ UC-05: プロジェクト論理削除
      └─ UC-06: プロジェクト検索（将来）

3. Service境界決定 ✅
   └─ 単一Service (ProjectsService) で完結
      └─ メソッド数: 5-6（適切な範囲）

4. Repository特定 ✅
   └─ ProjectsRepository
      └─ 1テーブル = 1Repositoryの原則

5. 依存関係整理 ✅
   └─ Service → Repository → DB
      └─ 単方向依存、循環依存なし

【ORM環境での原則】

汎用CRUD層 --> 作らない ❌
  └─ Drizzle ORMが提供する機能で十分
  └─ 過剰な抽象化を避ける

ドメイン固有メソッド --> 作る ✅
  ├─ getNextProjectNumber() - PME番号生成ロジック
  └─ mapToProjectTable() - データ変換ロジック
  └─ ビジネス意図を明確に表現
  └─ 複雑性の隠蔽
  └─ 再利用性・保守性の向上
```
